## 2. 핵심 비즈니스 로직
- **어드민 인증**: 비밀번호 `012560`을 DB 해시와 대조(verify_admin_password RPC)하며, 일치 시 서버사이드 관리자 계정 세션을 맺어 RLS를 우회합니다.
- **주문 및 미송(이월)**: 
  - 주문 포장완료 처리 시, 일부 품목만 체크되어 있으면 체크 완료된 품목은 `'포장완료'`, 미체크 품목은 `'미송'`으로 변경 후 신규 미송 주문서(`misong_orders`)로 자동 이월합니다.
  - 포장완료 취소(롤백) 시에는 생성되었던 미송 주문서가 자동 삭제되고 기존 주문 품목은 `'미포장'`으로 복구됩니다.
- **합배송 택배비 면제**: 한국 시각(KST) 새벽 4시 기준, 당일 영업 주기 내에 재주문(전화번호 동일) 건이 존재하면 택배 배송 방식을 골라도 배송비를 0원(무료)으로 자동 처리합니다.
- **전화번호 선(先)조회 강제화**: 주문서 작성 시 전화번호 조회 완료 전에는 하위 주소 및 결제 폼 전체를 화면에서 숨기며, 조회 후 전화번호를 수정하면 입력 폼 전체가 초기화 및 재은닉됩니다.

## 3. 웹 푸시(Web Push) 시스템
- **동작 방식**: 브라우저 표준 Web Push Protocol + Supabase Edge Function (`send-push`) + DB Webhook 연동.
- **기기 구독 동기화**: 
  - 고객 주문 완료 시점, 주문 조회 성공 시점, 어드민 로그인 성공 시점에 브라우저 토큰을 발급받아 `push_subscriptions` 테이블에 `upsert`합니다.
  - 사용자가 알림 동의를 해제하거나 로그아웃하면 해당 기기의 구독 정보를 DB에서 즉시 삭제합니다.
- **백그라운드 전송**: DB 테이블(`orders`, `misong_orders`)에 INSERT/UPDATE가 일어날 때 Webhook 트리거가 Deno Edge Function을 호출하여, 오프라인 상태인 기기 OS로 직접 알림을 쏩니다. 만료된 토큰(410 에러 등)은 Edge Function에서 즉시 감지하여 DB에서 자동 제거합니다.

## 4. UI/UX 및 스타일링 규칙
- **균형 있는 레이아웃 및 가독성**: 사용 연령층을 고려하되, UI 요소가 지나치게 비대해지지 않도록 컴팩트하고 깔끔하게 정돈된 크기(합리적인 폰트 크기 및 패딩)를 유지해야 합니다.
- **전체 배경 이미지와 가독성 분리**:
  - `body::before` 가상 요소를 생성하여 전체 배경에 지정된 이미지를 배치하고 `filter: blur(10px)`와 `transform: scale(1.05)`를 할당하여 사이트 내부 텍스트 및 UI 요소들과 시각적으로 완전히 분리합니다.
  - 글씨 가독성을 높이기 위해 유리 효과 불투명도 변수(`--glass-bg`)를 라이트/다크 모드 전반에서 높게 유지하고 글자 아웃라인(`text-shadow`) 효과를 적극 활용합니다.
- **모달 및 컨테이너 가독성 보장**:
  - 모바일 브라우저 등 `backdrop-filter`(블러 효과)가 완벽히 지원되지 않는 환경에서 뒷배경 글자가 모달을 뚫고 겹쳐 보여 눈 피로를 주는 현상을 해결하기 위해 모달 컨테이너(`.customer-modal-container`)는 `glassmorphism` 클래스 지정을 제거하고, `background: var(--bg) !important`를 지정하여 확실한 불투명 단색 배경으로 구현해야 합니다.
  - 개별 거래 내역 리스트 행(`.transaction-item-row`)이나 상세 영수증 카드(`.transaction-receipt-card`) 등도 투명 배경을 배제하고 `var(--bg)` 등의 불투명 배경을 명시적으로 깔아 주어야 합니다.
- **버튼 및 컨테이너의 시각적 구분감**:
  - 경계가 흐릿해 보이지 않도록 일반 상태의 모달 카드 테두리를 `1.5px` 이상으로 상향하고, 액션 버튼(상세보기, 사진 버튼 등)은 `2px` 두께의 실선 테두리와 명확한 그림자(`box-shadow`) 효과를 할당해 입체적인 구분감을 제공해야 합니다.
- **상태별 전용 색상 및 드롭다운 크기**:
  - 주문: 주황색 (`#d97706` / `rgba(245, 158, 11, 0.15)`)
  - 입금 대기중: 노란색 (`#ca8a04` / `rgba(234, 179, 8, 0.15)`)
  - 포장완료: 초록색 (`#10b981` / `rgba(16, 185, 129, 0.15)`)
  - 주문 취소: 회색 (`#9ca3af` / `rgba(156, 163, 175, 0.15)`)
  - 드롭다운 크기는 텍스트 길이에 맞춰 유동적으로 변화해야 합니다.
- **품절 및 임시 품절 상태 및 노출 로직**:
  - 관리 페이지의 상품 및 옵션 기능에서 '삭제' 기능은 '품절'로, '숨기기/노출중단' 기능은 '임시 품절'로 명칭을 통일합니다.
  - 품절(`is_deleted = true`) 처리된 상품은 쇼핑몰 화면에 노출되나, 흐릿하게 처리되고 대각선 줄이 가며 '품절'로 표시됩니다. 상세보기 화면은 제공하되 장바구니 수량 증가(+) 버튼이 비활성화되어 주문이 불가능합니다.
  - 임시 품절(`is_visible = false`) 처리된 상품 및 옵션도 쇼핑몰 화면에 노출되나, 흐릿하게 표시되고 '일시품절'로 명시됩니다. 마찬가지로 상세 보기 내에서 주문(장바구니 추가) 기능이 비활성화됩니다.
- **컴포넌트 일관성**: 접이식 상세 보기 버튼은 항상 타원형 캡슐 형태(`border-radius: 20px`)로 통일하고, 주문 카드는 바 영역 전체 클릭으로 토글되도록 설계합니다.
- **중복 제출 방지**: 주문 생성이나 상태 전환 등 DB 데이터의 INSERT/UPDATE를 유발하는 중요한 액션 버튼은 제출 프로세스가 수행되는 동안 버튼을 `disabled` 시켜 연속 클릭(더블 탭)을 방지하고 로딩 표시를 제공해야 합니다.

## 5. 에이전트 행동 및 커뮤니케이션 규칙
- **이모티콘(이모지) 사용 금지**: 에이전트는 사용자와 대화(답변)할 때나 보고서, 설명 문서 등을 작성할 때 불필요한 이모티콘을 사용하지 않고, 담백하고 차분한 명조 또는 고딕조 톤의 텍스트로 정보만 명확히 전달해야 합니다.
- **검증 제한 및 빌드 테스트만 실행**: 에이전트는 구현된 변경 사항에 대한 직접적인 기능/시나리오 검증(QA, 수동 동작 확인 등)을 수행해서는 안 되며, 오직 빌드 및 컴파일 테스트(`npm run build` 등)만 완료한 뒤 변경 사항을 보고해야 합니다. 실제 기능 작동 검증은 사용자가 직접 수행할 수 있도록 검증 시나리오와 확인 순서만을 명확히 제공해야 합니다.

## 6. 배포 환경 및 인프라
- **웹 호스팅 및 배포 (Cloudflare Workers Assets)**: 
  - 깃허브(GitHub) 저장소와 연동되어 커밋 푸시 시 자동으로 통합 배포되며, 최종 배포 도메인은 기존과 동일하게 mindle-order.ssrabw.workers.dev를 유지합니다. 
  - 로컬 환경 및 CLI 배포 시에는 wrangler deploy --config dist/mindle_order/wrangler.json 또는 wrangler dev --config dist/mindle_order/wrangler.json을 사용하여 빌드된 최적화 자산을 배포 및 가동합니다.
- **통합 서버리스 API 진입점 (src/worker.ts)**: 
  - 관리자 자격 증명 검증 및 ImgBB API Key 등 보안에 민감한 백엔드 정보들은 src/worker.ts 내의 서버리스 API 핸들러에서 안전하게 처리되며, 리액트 정적 자산 서빙과 결합되어 동작합니다. 로컬 개발 환경 구동 시에는 .dev.vars에 정의된 환경 변수들을 로드하여 사용합니다.
- **환경 변수 대시보드 등록 규칙 (중요)**:
  - 런타임 환경에서 백엔드 코드가 값들을 정상적으로 조회할 수 있도록, 반드시 Cloudflare Workers 설정의 **`변수 및 암호` (최상단 메뉴)** 페이지에 환경 변수들을 등록하고 수동 배포를 반영해야 합니다.
  - 설정 내 `빌드` 하위에 있는 변수 테이블에 등록하면 빌드 시점에만 유효하고 구동(런타임) 시점에는 `undefined`가 되므로 오동작하게 됩니다.
- **데이터베이스 및 백엔드 서비스 (Supabase)**: 
  - Supabase PostgreSQL을 사용합니다. 관리자 로그인 시 서버리스 API(/api/admin/login)가 내부 환경 변수를 통해 인증을 검증하고 세션 객체를 돌려줍니다. 클라이언트는 이 세션을 supabase.auth.setSession으로 주입받아 RLS 우회 권한을 획득합니다. 이때 API Key 불일치 에러를 차단하기 위해 서버사이드 클라이언트 생성 시에도 VITE_SUPABASE_PUBLISHABLE_KEY를 사용해야 합니다.
- **이미지 저장소 및 실시간 압축 (ImgBB)**: 
  - 이미지 영구 호스팅에 ImgBB API를 사용합니다. 클라이언트는 API Key 노출 없이 서버리스 프록시 API(/api/image/upload)를 호출하여 업로드합니다.
  - 고용량 이미지 업로드로 인한 부하 및 속도 저하를 방지하기 위해, 클라이언트 단(AdminPage.tsx의 uploadToImgBB 및 compressImage)에서 Canvas API를 사용하여 해상도 최대 크기를 1200px(종횡비 유지)로 제한하고, JPEG 75% 화질로 실시간 압축 변환한 뒤 업로드하도록 설계되어 있습니다.

## 7. 카테고리 관리 동작 규칙

## 8. 베스트 설정(Best Product) 관리 및 노출 규칙
- **어드민 베스트 설정 모달**:
  - 어드민 상품 관리 페이지 탭 바 내에서 '상품 관리', '상품 등록' 탭의 바로 뒤에 순서대로 배치된 '베스트 설정' 버튼을 클릭하면 베스트 지정 모달이 생성됩니다. 등록된 상품 목록 중 활성 상태의 상품들만 필터링하여 노출하며, 토글 스위치 방식으로 DB의 `is_best` 플래그 값을 즉각 저장 및 동기화합니다.
  - UI 일관성을 위해 기존의 '베스트 상품 설정' 표기를 모두 '베스트 설정'으로 일관되게 변경하였습니다.
- **메인 베스트 쇼케이스 레이아웃**:
  - 활성화된 베스트 상품들(`isBest && !isDeleted && isVisible`)을 대상으로 필터 바 상단에 전용 레이어를 노출합니다.
  - 반응형 슬라이드 연동: 모바일 환경에서는 베스트 상품이 3개 이상일 때, PC 환경에서는 5개 이상일 때 무한 루프 슬라이드 마르퀴(Marquee) 애니메이션을 가동하여 부드럽고 천천히 좌측으로 흘러가도록 노출합니다. 마우스 호버(Hover) 시 슬라이드가 일시정지됩니다.
  - 모바일이나 PC의 각 조건 개수 미만일 때는 움직이지 않고 가로 방향 중앙 정렬 정적 카드로 렌더링됩니다.
  - 베스트 상품 카드의 이미지 크기(너비)는 PC 190px, 모바일 125px로 일반 상품 카드보다 약간만 더 큰 수준으로 고급화하고, 카드 내부의 여백 및 줄바꿈 공간 낭비를 방지하기 위해 제목과 가격의 패딩을 축소(8px) 및 높이 auto 처리하였습니다.
  - 슬라이더 좌우 끝 그라데이션 페이드 영역을 3%로 축소하여 상품 노출 선명도를 향상시켰습니다.
- **Best 배지 및 상태 배지 중첩 제어**:
  - 베스트 상품은 이미지 왼쪽 상단에 입체적인 엠버 골드 그라데이션 컬러의 `Best` 표시 배지가 오버레이됩니다.
  - 베스트 상품이면서 품절/일시품절 처리된 경우, 품절 안내 배지는 우측 상단으로 자동 이동(오른쪽 정렬)하여 배지 겹침을 방지합니다.

## 9. 상품 완전 삭제(Real Delete) 규칙
- **완전 삭제 상태 (`is_real_deleted === true`)**:
  - 상품이 실수로 잘못 업로드된 경우, 데이터베이스 레코드는 보존하되 판매 페이지(쇼핑몰 목록 및 베스트 레이어, 상세 보기 화면) 전체에서 완전히 격리 은닉하기 위해 사용됩니다.
  - 완전 삭제된 상품은 쇼핑몰 화면에 노출되지 않으며, URL을 통해 직접 상세 페이지에 진입을 시도할 경우에도 상품을 찾을 수 없는 상태로 처리됩니다.
- **어드민 삭제 상품 관리 탭**:
  - 관리자 대시보드 상품 관리 섹션에 '삭제된 상품' 탭이 추가되었습니다. 이 탭에서는 완전 삭제 처리된 상품들만 모아서 확인할 수 있습니다.
  - 관리자 페이지의 각 상품 카드 하단에서 '완전 삭제' 버튼을 클릭하면 해당 상품은 '삭제된 상품' 탭으로 이동하게 됩니다.
  - '삭제된 상품' 탭 내부에서는 상품 정보나 색상 옵션을 더 이상 수정할 수 없으며, 오직 '복원하기' 기능만 사용 가능합니다. 복원 시 상품은 이전의 품절 여부 및 활성 상태를 그대로 유지한 채 기존 탭으로 돌아갑니다.
